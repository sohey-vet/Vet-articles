# Note自動下書き投稿ワークフロー

ユーザーから「Noteの下書きを作って」「Noteにアップして」と指示された際、以下の手順で進めてください。

## 0. 事前確認（Markdownテンプレートの構造）
まず対象となる記事のMarkdownファイルが、以下の「Noteに最適化されたひな形」に準拠しているか確認・修正してください。
特に、記事末尾の**「飼い主への説明ガイド」「参照論文」**が抜け落ちていないか必ずチェックし、無ければ追加しておくこと。
（※HTMLで用意されている元データである『ゴールドスタンダード』をそのまま生かす方針です）

**【超重要：Note用自動投稿スクリプトの仕様と教訓（2026-04 最新）】**
今後、Noteへの下書き投稿システム（`auto_post_note.py`）に関連するコードを改修する機会があった場合でも、以下の**完成された仕様**を絶対に破壊しないこと（AIの誤った推測による改悪を禁ず）。
1. **カードスタイル・リスト化最適機構**: Note側のブロックエディタは外部から注入された `<table`> タグを正常に解釈できず、表示を崩壊させる。そのため、このスクリプト内にはMarkdownの表を意図的に解体し、`<strong>【項目：値】</strong> / ・見出し２：値` の美しい段落に変換する（旧来の手動パースによるアナログHTML生成）という**唯一の最適解**が実装されている。これを「バグだ」と誤認して撤去したり、勝手に `markdown` モジュールによるタグ生成に書き換えたりすることは厳禁である。
2. **参照文献数の自動カウント機能**: 冒頭の「読了時間」の直下に、「📚 参照文献: N本」というテキストを挿入するため、スクリプトは原稿の末尾にある `## 📚 参照論文` 配下のリスト行数を自動パースしている。Markdownには必ず参照論文リストを残しておくこと（以前のように削除してはならない）。
3. **UI競合バグの回避**: Playwright操作中、タイトル入力直後に送信される `Escape` キーや、本文フォーカスまでの待機時間（sleep）は、Noteエディタ特有のサジェスト等による「タイトルへの本文混入バグ」を防ぐコア部分であるため、コードを軽率に削らないこと。

```markdown
# 🐕 タイトル
> ⏱️ **読了時間**: 約X分

## 🎯 結論
（テキスト）

## 🗺️ 見出し
| 表のヘッダー1 | 表のヘッダー2 |
|---|---|
| 行 | 行 |

### ▼ 小見出し
（テキスト）

**臨床アクション** （※スクリプト側で自動的に💡が付与されるため `💡 臨床アクション` と書いても可）
（アクション内容）

## 🗣️ 飼い主への説明ガイド
（Q&Aや説明文など）

## 📚 参照論文
（箇条書き）
```

## 1. 自動投稿スクリプトの準備と実行（--draft 必須）
対象となる記事の「公開予定日（例：2026-04-01）」を `sns_schedule.json` もしくは指示から特定し、以下のコマンドで実行してください。
⚠️ **必ず `--draft` フラグを付与して下書きとして保存し、いきなり公開しないこと。**

```bash
cd c:\Users\souhe\Desktop\VetEvidence_SNS_Drafts\VetEvidence_AutoPost
python auto_post_note.py --date "YYYY-MM-DD" --draft
```

## 2. スクリプトの実行と監視
このスクリプトはPlaywrightを使用してブラウザを操作し、以下の処理を自動で行ってくれます。
1. Noteの執筆画面を自動で開く
2. サムネイル画像（note_thumbnail.png）のアップロードとクロップモーダルの突破
3. Markdownから変換されたHTMLのクリップボード経由でのペースト（レイアウト保持）
4. タイトルやハッシュタグは入力せず、そのまま下書き保存

※実行後、`コマンドID`のステータスを追跡し、`Status: DONE` なしには「✅ 下書きとして完全に保存されました」のログが出力されるまで待機してください（通常約30秒〜1分）。

## 3. 結果の報告
実行完了後、Noteの管理画面に記事が正しく下書き生成された旨をユーザーに報告してください。その際、💡アイコンや記事末尾のセクションが反映されているはずである点にも言及すると親切です。
